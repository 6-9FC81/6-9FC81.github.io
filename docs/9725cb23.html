面向对象设计原则

依赖倒置原则(DIP)

  • 高层模块(稳定)不应该依赖于低层模块(变化)，二者都应该依赖于抽象(稳定)
    • 举个栗子，现有高层模块(mainform)，它依赖多个低层模块(line、rect)。其中，高层模块(mainform)是一个单一的功能，我们认为它是稳定的。低层模块(line、rect)是高层模块(mainform)的组成部分，这部分代码有可能会发生变化，因此它是不稳定的。
    • 稳定的高层去依赖不稳定的底层，这显然会有问题。因此应当新建一个抽象(shape)，将不稳定的、会变化的部分整合起来，使高层模块(mainform)依赖抽象(shape)，而低层模块(line、rect)也同时依赖抽象(shape)
    • 这样做可以隔离变化，使原来的高层模块(mainform)和层模块(line、rect)都依赖一个稳定的部分，于是变化被隔离。
  • 抽象(稳定)不应该依赖于实现细节(变化)，实现细节应该依赖于抽象(稳定)
    • 抽象类中不应该使用具体实现的子类
  • 一句话总结：底层依赖必须是稳定的，如果做不到稳定就使用抽象类来隔离变化

开闭原则(OCP)

  • 对拓展开放，对更改封闭。
  • 类模块应该是可拓展的、不可修改的
  • 一句话总结：使用拓展的方式来应对需求变化，而不是修改。

单一职责原则(SRP)

  • 一个类应该仅有一个引起它变化的原因
  • 变化的方向隐含着类的责任
  • 一句话总结：一个类只干一件事。目的：便于理解，提高代码可读性

里氏替换原则(LSP)

  • 子类必须能替换它们的基类(子类能使用父类的所有方法)
  • 继承表达类型抽象
  • 典型反例：父类：鸟，子类：鸵鸟。继承完了之后你发现父类里有个Fly方法，但鸵鸟不会飞。于是你重载Fly方法，子类调用此方法就返回错误。这显然违背了这一原则
  • 一句话总结：切勿继承滥用继承

组合优先原则

  • 优先使用对象组合，而不是类继承
  • 类继承通常为“白箱复用”，对象组合通常为“黑箱复用”。
  • 继承在某种程度上破坏了封装性，子类父类耦合度高。
  • 对象组合只要求被组合的对象具有良好定义的外部接口，耦合度低。
  • 一句话总结：多用组合，少用继承

接口隔离原则(ISP)

  • 不应该强迫客户程序依赖它们不用的方法
  • 接口应该小而完备
  • 一句话总结：慎用public，尽量private。

迪米特原则(LOD)

  • 使用封装来创建对象之间的分界层，让设计者可以在分界层的一侧进行修改，而不会被另一侧造成不良的影响
  • 一句话总结：封装变化点，一侧稳定一侧变化。

针对接口编程，而不是针对实现编程

  • 不将变量类型声明为某个特定的具体类，而是声明为某个接口(针对业务类型)

  • 客户程序无需获知对象的具体类型，只需知道对象所具有的接口

  • 这个原则与依赖倒置原则相辅相成，从不同角度看待同一个问题，违背一个往往同时违背了另一个

  • 接口标准化！

重构关键技法

  • 静态绑定 -> 动态绑定
  • 早绑定 -> 晚绑定
  • 继承 -> 组合
  • 编译时依赖 -> 运行时依赖
  • 高耦合 -> 低耦合

组件协作模式

  • 现代软件专业分工后的第一个结果是“框架与应用的划分”，“组件协作”模式通过晚绑定来实现框架与应用程序之间的松耦合，是二者之间协作时常用的模式。
  • 典型模式
    • Template Method 模板方法
    • Strategy 策略模式
    • Observer / Event 观察者/事件模式

模板方法 Template Method

应用背景

  • 在软件构建过程中，对于某一项任务，它常常有稳定的整体操作流程，但各个子步骤却有很多改变的需求，或者由于固有的原因(比如框架与应用之间)而无法框架和任务的整体结构同时实现。
  • 如何在确定稳定操作结构的前提下，来灵活应对各个子步骤的变化或者晚期实现需求？

模式定义

定义一个操作中的算法的骨架**(稳定)，而将一些步骤(变化的部分)**延迟到子类中。Template Method 使得子类可以不改变(复用)一个算法的结构即可重定义(override 重写)该算法中某些特定步骤。

总结

  • 你必须能分辨出哪些部分是稳定的部分，哪些部分是变化的部分。
  • 必须有一个相对稳定的算法整体流程可以重用。
  • 该模式是一种非常基础的设计模式。它用最简洁的机制(虚函数的多态性)为很多应用程序架构提供了灵活的拓展点，是代码复用方面的基础实现结构
  • 除了可以灵活应对子步骤的变化外，“不要调用我，让我来调用你”的反向控制结构是该模式的典型应用。
  • 在具体实现方面，被调用的虚方法可以具有实现，也可以没有任何实现(抽象方法、纯虚方法)，但一般推荐将它们设置为 protected 方法。

策略模式 Strategy

应用背景

  • 在软件构建过程中，某些对象使用的算法可能多种多样，经常改变，如果将这些算法都编码到对象中，将会使对象变得异常复杂；而且有时候支持不使用的算法也是一个性能负担。
  • 如何在运行时根据需要透明地更改对象的算法?将算法与对象本身解耦，从而避免上述问题？

模式定义

定义一系列算法，把它们一个个封装起来，并且使它们可互相替换**(变化)。该模式使得算法可独立于使用它的客户程序(稳定)**而变化(拓展、子类化)。

总结

  • Strategy 及其子类为组件提供了一系列可重用的算法，从而可以使得类型在运行时方便的根据需要在各个算法之间进行切换。
  • Strategy 模式提供了条件判断语句以外的另一种选择，消除条件判断语句，就是在解耦合。含有许多条件判断语句的代码通常都需要 Strategy 模式。
  • 如果 Strategy 对象没有实例变量，那么各个上下文可以共享同一个 Strategy 对象，从而节省对象开销。
  • 关于复用性：我们一般认为复用是二进制代码级别的复用性，编译、测试、部署后可以原封不动的复用。而不是代码片段复用，你把一份代码复制粘贴很多份那不叫复用。
  • 关于”支持不使用的算法也是一个性能负担”：譬如本例是以国家举例，那么假如程序部署到某一个国家，那么基本不存在进入另外两个条件的可能。但写法一仍会将另外两种情况的代码包含在程序里面，这些代码段最终都要被装载到 cpu 的高级缓存里，假如 cpu 内存不够甚至还要用到虚拟内存，而写法二可以缓解这个问题。当然这个只是顺带的好处，不是 Strategy 设计模式真正要解决的问题。

观察者模式 Observer / Event

应用背景

在软件构建过程中，我们需要为某些对象建立一种“通知依赖关系”：一个对象(目标对象)的状态发生改变，所有的依赖对象(观察者对象)都将得到通知。如果这样的依赖关系过于紧密，将使软件不能很好的抵御变化。

模式定义

定义对象间的一种一对多(变化)的依赖关系，以便当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并自动更新。

总结

  • 观察者模式可以有很多种方式实现，但最关键的还是抽象的通知依赖关系
  • 使用面向对象的抽象，观察者模式使得我们可以独立地改变目标与观察者，从而使二者之间的依赖关系松耦合
  • 目标发送通知时，无需指定观察者，通知(可以携带通知信息作为参数)会自动传播。
  • 观察者自己决定是否需要订阅通知，目标对象对此一无所知。
  • 观察者模式是基于事件的UI框架中非常常用的设计模式，也是MVC模式的一个重要组成部分
  • 关于依赖：默认指编译时依赖，即被依赖的东西必须存在才能编译通过。与之相对的是运行时依赖，需要运行时存在。

单一职责模式

  • 在软件组件设计中，如果责任划分的不清晰，使用继承得到的结果往往是随着需求的变化，子类急剧膨胀，同时充斥着重复代码，这时候的关键是划清责任。
  • 典型模式
    • Decorator 装饰模式
    • Bridge

装饰模式 Decorator

应用背景

  • 在某些情况下我们可能会”过度地使用继承来扩展对象的功能”，由于继承为类型引入的静态特质，使得这种扩展方式缺乏灵活性；并且随着子类的增多(扩展功能的增多)，各种子类的组合(扩展功能的组合)会导致更多子类的膨胀。
  • 如何使”对象功能的扩展”能够根据需要来动态地实现？同时避”扩展功能的增多”带来的子类膨胀问题？从而使得任何”功能扩展变化”所导致的影响降为最低？
